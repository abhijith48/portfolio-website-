04 Method 

    
How I work the problem.

    

      

        
01

        
Define "good" first

        
Before touching a prompt, write the rubric. What does a correct, grounded, useful response look like, and what exactly counts as failure? At NVIDIA this meant evaluation rubrics and QA checklists before optimization, never after.

      

      

        
02

        
Build the test set

        
Real inputs, edge cases, adversarial phrasings. A prompt that only works on the happy path doesn't work. Every project I ship carries its test cases with it: in YAML, versioned, next to the prompt.

      

      

        
03

        
Run, score, compare

        
Variants get A/B tested against the rubric: accuracy, hallucination rate, relevance, consistency. Numbers over vibes. A prompt change without a measured delta is a guess, not an improvement.

      

      

        
04

        
Iterate on the failures

        
Study the misses (hallucination, reasoning gaps, format drift) and patch the prompt for the failure mode, not the symptom. Re-run, re-score, and repeat until the metrics hold under pressure.

      

    


    

      
The delta, made visible · one pattern from my eval workflow

      

        

          
Variant A · baseline

          
Summarize the following support ticket.

          

            accuracy 0.83
            hallucination 0.06
            format drift 22%
          

        

        

          
Variant B · shipped

          
You are a support triage assistant.
Summarize the ticket below in exactly
3 bullets: issue, impact, requested action.
If a detail is missing, write "not stated".
Never infer it.

          

            accuracy 0.94 (+0.11)
            hallucination 0.00 (−0.06)
            format drift 0%
          

        

      

      
The difference isn't cleverness. It's constraints: a role, a fixed structure, and an explicit out for missing information.
